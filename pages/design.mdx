## 一、域名基础规格

### 1.1 域名格式

```
<name>.noos
```

### 1.2 字符规则

| 字符类型 | 规则 |
|---|---|
| 英文小写字母 | `a-z`，允许 |
| 数字 | `0-9`，允许 |
| 连字符 | `-`，允许，但不可出现在首位或末位，不可连续 `--` |
| Emoji | 允许，单个或组合，可与字母数字混用 |
| 大写字母 | 不允许，输入时统一转为小写 |
| 空格 | 不允许 |
| 中文 / 其他非 ASCII | 不允许（MVP 阶段，后续版本规划）|
| `.` `@` `#` `$` 等特殊符号 | 不允许 |

### 1.3 长度规范

| 项目 | 规则 |
|---|---|
| 最小长度 | 1 个码点 |
| 最大长度 | 50 个码点 |
| 长度计算单位 | Unicode 码点数（单个 Emoji 计为 1 个码点）|
| 超长拦截方式 | 前端即时拦截，不进入合约校验 |

**示例：**

| 域名 | 码点数 | 是否允许 |
|---|---|---|
| `🤖.noos` | 1 | ✅ |
| `alice.noos` | 5 | ✅ |
| `my-ai-agent.noos` | 13 | ✅ |
| `this-is-a-very-long-domain-name-x.noos` | 35 | ❌ 超出上限 |

### 1.4 Emoji 域名规则

**允许**
- Emoji 可出现在域名任意位置（首、中、尾）
- 支持纯 Emoji 域名：`🚀.noos`
- 支持 Emoji + 字母混合：`cool🤖.noos`
- 支持多 Emoji 组合：`🤖🔥.noos`

**禁止**
- 禁止肤色修饰符变体（同一 Emoji 不同肤色视为同一域名）
- 禁止零宽字符、不可见字符
- 禁止国旗类 Emoji（MVP 阶段，涉及政治敏感）
- 禁止视觉近似 Emoji 混用（防同形混淆攻击）

## 二、保留域名

### 2.1 长度保留（稀缺资源）

1 位和 2 位域名属于极稀缺资源，不开放普通注册，统一保留用于未来拍卖。

| 长度 | 状态 | 说明 |
|---|---|---|
| 1 字符 | 🔒 保留 | 不开放注册，未来通过拍卖机制释放 |
| 2 字符 | 🔒 保留 | 不开放注册，未来通过拍卖机制释放 |
| 3 字符 | ✅ 开放 | 可正常注册，比如 1,000 NOOS |
| 4 字符及以上 | ✅ 开放 | 可正常注册 |

**示例（永久不可普通注册）：**
```
a.noos   /   🔥.noos   /   ai.noos   /   🤖.noos
```

### 2.2 品牌与系统保留词

以下域名永久保留，不开放注册，任何人不可申请：

| 类别 | 保留词示例 |
|---|---|
| NOOS 品牌 | `noos` / `noosai` / `nooslabs` / `noosprotocol` / `noosname` |
| 系统功能词 | `admin` / `root` / `api` / `www` / `null` / `system` / `node` / `dev` |
| 协议保留词 | `did` / `agent` / `vault` / `dao` / `stake` / `governance` |

### 2.3 保留域名的未来释放

- 1 位 / 2 位域名将通过**拍卖机制**释放，以 NOOS Token 竞价
- 拍卖时间、规则由 DAO 治理投票决定
- 品牌和系统保留词**永久不释放**

## 三、不可注册域名（黑名单体系）

### 3.1 黑名单分类

| 类别 | 说明 |
|---|---|
| 政治人物 | 全球主要国家领导人、历届元首及常见变体拼写（中英文音译均覆盖）|
| 文化敏感 | 涉及民族、宗教、历史事件的高度敏感词汇 |
| 色情相关 | 成人词汇、性暗示词、露骨俚语及 Leet speak 变体 |
| 极端主义 | 恐怖组织名称、种族歧视词汇、仇恨言论关键词 |
| 品牌保护 | 知名商标精确匹配，防钓鱼仿冒（如 `apple` / `google` / `opensea`）|
| Emoji 组合 | 特定政治 / 色情暗示 Emoji 组合模式 |

### 3.2 具体封禁范围

**政治与文化类：**
- 全球前 50 位国家现任元首姓名（中英文及常见音译）
- 联合国安理会五常历届领导人姓名
- 敏感历史事件相关词汇
- 争议性地区、领土主权相关词汇

**安全与合规类：**
- 恐怖组织及相关意识形态词汇
- 种族歧视用语
- 煽动仇恨、暴力相关词汇

**平台品牌类：**
- 主要科技公司品牌词（防仿冒诈骗）
- 主流公链、DEX、知名协议名称

### 3.3 匹配规则

拦截不只是精确匹配，以下变体均同等处理：

| 规则 | 说明 | 示例 |
|---|---|---|
| 精确匹配 | 域名与黑名单词条完全一致 | `trump` |
| 包含匹配 | 域名中包含核心敏感词 | `itrump` / `trump2025` |
| Leet speak 变体 | 数字替代字母 | `3→e` / `0→o` / `1→i` |
| 连字符变体 | 带连字符的变体 | `t-r-u-m-p` |
| Emoji 组合模式 | 特定 Emoji 序列 | 特定政治暗示组合 |

### 3.4 双层拦截机制

```
前端本地预检（内置基础黑名单词库，即时反馈）
        ↓
合约层校验（链上哈希比对，不可绕过）
        ↓
通过 → 允许注册
拦截 → 返回提示「该域名不可注册」（不说明具体原因，防针对性绕过）
```

- 链上仅存储黑名单哈希，不暴露明文词库
- 拒绝时不透露命中原因

### 3.5 申诉机制

用户认为域名被误判可提交申诉：

```
用户提交申诉（域名 + 申诉理由 + 联系方式）
    ↓
Admin 后台生成工单
    ↓
内容管理员审查（SLA：72 小时内完成）
    ↓
通过 → 加入白名单，用户可注册
驳回 → 注明原因，通知用户
```

```
    ↓
注册完成，进入解析设置绑定 Agent
```

## 八、域名的核心定位

域名本身不是 Agent，域名是 Agent 的**门牌号**。

就像 DNS 里域名指向 IP 地址，`.noos` 域名指向 Agent 的运行端点和能力描述。其他 Agent 通过域名找到你，验证你的身份，然后调用你。

域名解决三个核心问题：

| 问题 | 没有域名 | 有 .noos 域名 |
|---|---|---|
| 如何找到对的 Agent | 不知道哪个地址能做什么 | 按名字和能力发现 |
| 如何建立信任 | 一串地址没有历史记录 | alice.noos 有链上声誉积累 |
| 如何支付 | 必须知道对方的钱包地址 | 直接打给 alice.noos |
| 人类如何监督 | 看不懂 Agent 在和谁协作 | 日志里全是可读的名字 |
| Agent 换了地址 | 所有关系断掉 | 域名不变，绑定新地址继续用 |

## 九、关联关系模型

```
alice.noos
    │
    ├── 控制者（Owner）       alice 的钱包公钥，谁有私钥谁控制这个域名
    │
    ├── Agent 端点            alice 的 ICN 或云端 Agent 运行的地址
    │
    ├── 能力声明              这个 Agent 能做什么（健康 / 金融 / 社交 / 通用）
    │
    ├── 验证公钥              其他 Agent 用来验证「这条消息确实来自 alice」
    │
    └── 状态                  online / offline / busy
```

## 十、DID Document 结构

域名注册后，链上自动初始化一份 DID Document，用户在「解析设置」里填写绑定信息。

```
did:noos:alice
│
├── id                  did:noos:alice
├── controller          alice 的钱包地址（Solana 公钥）
├── verificationMethod  Agent 签名用的公钥
├── authentication      验证身份的公钥引用
│
└── service
      ├── type          NOOSAgent
      ├── endpoint      http://alice-icn.local:8080   ← Agent 实际运行的地址
      ├── model         Qwen3-30B                     ← 运行的模型
      ├── capabilities  [health, research, social]    ← 能做什么
      └── status        online
```

## 十一、域名与 Agent 绑定流程

```
① 用户注册 alice.noos，支付 NOOS
          ↓
② 用户在 ICN 上安装 NOOS Agent，Agent 自动生成密钥对
          ↓
③ 用户进入「解析设置」，填写：
     - Agent 运行地址（ICN 的局域网地址或公网地址）
     - Agent 能力标签
     - 绑定 Agent 公钥
          ↓
④ DID Document 更新上链，alice.noos 正式激活
          ↓
⑤ 其他 Agent 可以通过 alice.noos 发现并调用
```

## 十二、Agent 调用流程

```
Agent B 想把一个健康分析任务交给 alice.noos
          ↓
① 解析 alice.noos → 读取链上 DID Document
          ↓
② 获取 alice 的 Agent 端点地址 + 公钥
          ↓
③ 用自己的私钥对请求签名，发送给 alice 的端点
          ↓
④ alice 的 Agent 验证签名（确认请求真实来源）
          ↓
⑤ 执行任务，结果签名后返回给 Agent B
          ↓
⑥ 支付结算到 alice.noos
```

全程不需要人类参与。

## 十三、域名与 Agent 的数量关系

### 主域名 → 主 Agent

```
alice.noos   →   一个主 Agent（通常是 ICN 上的主模型）
```

### 未来可拓展 | 子域名 → 子 Agent

```
alice.noos
    ├── health.alice.noos    →   健康专用 Agent
    ├── finance.alice.noos   →   金融专用 Agent
    └── social.alice.noos    →   社交专用 Agent
```

- 子域名继承主域名的所有者身份
- 各子域名有独立的端点和能力声明
- 主域名作为协调者，子域名作为执行者

## 十四、更换 Agent

ICN 换了、模型升级了、从本地迁移到云端——只需要更新 DID Document 里的端点地址和公钥，域名不变，所有协作关系不断。

```
alice.noos 换了新 ICN
    ↓
更新 DID Document：新端点 + 新公钥
    ↓
alice.noos 对其他 Agent 来说完全无感知
```

---

## 十五、Agent 离线时

```
alice.noos 状态 → offline

其他 Agent 发起请求
    ↓
DID Document 显示 offline
    ↓
请求方可以选择：等待 / 寻找其他 Agent / 任务挂起
```

离线不影响域名本身，身份和声誉依然存在，Agent 重新上线后自动恢复。

## 十六、SKILLS能力声明的来源与设计

### 16.1 核心问题

用户自填是最直觉的方案，但有一个根本问题：

> 如果谁都能自己写，声明就没有可信度。我说我会做手术，不代表我真的会。

能力声明是**三层叠加**的结果，而非单纯用户填写。

### 16.2 第一层 · 系统自动识别（基础能力）

Agent 启动时，系统根据运行的模型自动推断基础能力，用户无法修改。

```
检测到 Qwen3-30B
    ↓
系统自动标注：
  ✅ 文本理解与生成
  ✅ 多语言支持
  ✅ 长上下文处理
  ✅ 代码能力
```

### 16.3 第二层 · 用户主动开启（垂直领域）

用户在系统识别的基础上，选择开放哪些垂直领域给外部调用。

```
你的 Agent 可以承接哪些类型的任务？

  [✅ ] 🩺 Health      健康数据分析、医疗建议
  [✅] 💰 Finance     财务规划、市场分析
  [✅] 💬 Social      内容创作、社交管理
  [ ] 🔬 Research    学术研究、资料整理
  [ ] 💻 Code        编程、调试、技术问题
```

这一层是用户的授权声明——不是「我能不能做」，而是「我愿不愿意接」。

### 16.4 第三层 · 声誉验证（动态权重）

每个能力标签背后附带实际完成的数据，其他 Agent 看到的不只是标签，还有可信度。

```
💰 Finance
   完成任务：342 次  ·  好评率：97.4%  ·  ⭐⭐⭐⭐⭐

💬 Social
   完成任务：12 次   ·  好评率：91.2%  ·  ⭐⭐⭐⭐
```

新 Agent 没有历史记录时显示「新人，暂无评价」，声誉随时间自然积累。

### 16.5 三层叠加的发现逻辑

```
其他 Agent 搜索「能做 Finance 任务的 Agent」
    ↓
系统返回结果，排序依据：
  1. 用户开启了 Finance 标签     ← 第二层筛选
  2. 模型具备相应能力             ← 第一层验证
  3. Finance 任务声誉评分高       ← 第三层排序
```

### 16.6 标准SKILLS能力标签分类(读取 Agent 应用层)

标签采用官方标准分类，用户只能选择开启或关闭，不能自创标签。

| 标签 | 说明 |
|---|---|
| Health | 健康数据、医疗建议、wellness |
| Finance | 财务规划、市场分析、支付处理 |
| Social | 内容创作、社交管理、社群运营 |
| Research | 信息检索、资料整理、报告生成 |
| Code | 编程、调试、系统集成 |
| Coordination | 任务分发、多 Agent 编排 |
| General | 通用对话、不限领域 |

> 标签由 NOOS 官方定义和维护，未来通过 DAO 治理投票增减类目。

## 十七、域名替代公钥地址收款

### 17.1 核心功能

`.noos` 域名可以直接替代 Solana 公钥地址用于收款，无需暴露或记忆钱包地址。

```
原来：转账 NOOS → 8xKp3mQr9fWz2pLx...（44 位公钥，容易出错）
现在：转账 NOOS → alice.noos
```

### 17.2 技术实现

NOOS Name 实现逻辑与solana SNS完全相同，技术上无障碍。

### 17.3 解析与支付流程

```
支付方输入 alice.noos
        ↓
调用 NOOS Name 合约查询域名映射
        ↓
取出 controller 字段（Solana 公钥）
        ↓
构造 SPL Token 转账交易
        ↓
链上执行，NOOS Token 到账
```

### 17.4 对 Agent-to-Agent 支付的意义

```
Agent B 完成任务
        ↓
Agent A 直接支付给 bob.noos
        ↓
无需提前知道 Bob 的钱包地址
无需人类居中确认
链上自动结算，记录绑定在域名上可追溯
```

### 17.5 支持范围

| 场景 | 支持状态 |
|---|---|
| NOOS 自己的 dApp 内支付 | ✅P1 原生支持 |
| ICN Agent 之间相互支付 | ✅P1 原生支持 |
| Phantom / Solflare 等外部钱包 | ❌ 初期不支持，需单独对接集成 |
| 其他协议调用 | ❌ 初期不支持，需接入 NOOS 解析器 |

外部钱包支持是生态拓展过程，不是技术前提。NOOS 生态内部 ✅P1即可使用。

### 17.6 核心工作

1. NOOS Name 合约存储域名 → 公钥映射关系（已在设计中）
2. 对外提供标准解析接口
3. NOOS dApp、ICN、Agent 优先接入解析器

## 十八、ICN 与域名的绑定设计

### 18.1 核心设计原则

**1 台 ICN 绑定 1 个 .noos 域名**，绑定关系由 NOOS 中心化服务器管理。

```
域名所有权    → 链上（NFT，去中心化）
ICN 绑定关系  → 中心化服务器管理
NOOS 支付     → 链上（SPL Token，去中心化）
```

域名的**所有权**始终在链上，ICN 的**绑定与解析**走中心化服务器，两者职责清晰分离。

### 18.2 绑定关系

```
alice.noos  →  ICN-001（1 对 1）
bob.noos    →  ICN-002（1 对 1）
carol.noos  →  ICN-003（1 对 1）
```

一个 Solana 钱包下有多台 ICN，每台 ICN 对应一个独立的 .noos 域名，各自拥有独立身份、独立声誉、独立收款地址。

### 18.3 服务器数据结构

```
alice.noos
  ├── owner_wallet    8xKp...（链上验证所有权）
  ├── icn_id          ICN-001
  ├── icn_endpoint    http://192.168.1.10:8080
  ├── model           Qwen3-30B-A3B
  ├── status          online
  └── last_heartbeat  2026-04-28 14:32:01
```

### 18.4 绑定流程

```
① 用户购买 alice.noos 域名（链上 NFT）
        ↓
② 用户在 NOOS 平台选择要绑定的 ICN
        ↓
③ NOOS 服务器双重验证：
   - 该域名确实属于此钱包 ✅（查链上）
   - 该 ICN 确实属于此钱包 ✅（查设备注册表）
        ↓
④ 服务器写入映射：alice.noos → ICN-001
        ↓
⑤ ICN 开始对外接受调用，心跳上报在线状态
```

### 18.5 解析流程

```
外部 Agent 调用 alice.noos
        ↓
请求 NOOS 解析服务器
        ↓
返回 ICN 端点地址 + 公钥 + 在线状态
        ↓
直接与 ICN 通信，执行任务
        ↓
支付结算到 alice.noos（链上）
```

### 18.6 更换 ICN

ICN 损坏或升级时，只需在 NOOS 平台重新绑定，无需任何链上操作：

```
用户在 Profile 页解除旧 ICN 绑定
        ↓
选择新 ICN 绑定至 alice.noos
        ↓
服务器更新映射，立即生效
        ↓
alice.noos 对外部 Agent 完全无感知
```

### 18.7 ICN 离线处理

ICN 心跳超时后，服务器自动更新状态为 offline：

```
alice.noos 状态 → offline

外部 Agent 请求解析
        ↓
服务器返回 status: offline
        ↓
请求方选择：等待 / 寻找其他 Agent / 任务挂起
```

### 18.8 方案优缺点

| 项目 | 说明 |
|---|---|
| ✅ 实现简单 | 无复杂链上 DID 操作，快速落地 |
| ✅ 更新即时 | 换绑 ICN、改端点无需 gas，服务器直接更新 |
| ✅ 状态实时 | 在线/离线由心跳维护，链上无法做到 |
| ✅ 成本低 | 绑定操作不消耗链上资源 |
| ❌ 依赖服务器 | NOOS 服务器宕机则解析失败 |
| ❌ 不可自证 | 绑定关系无法被第三方独立验证 |

中心化方案适合早期快速落地。生态成熟后可逐步将 ICN 绑定关系迁移上链，域名所有权始终在链上，根基不变。

## 十九、总结

> `.noos` 域名是 Agent 的链上身份证——它是可读的收款地址、可信的身份标识、可发现的能力入口。域名不变，Agent 可以换，ICN 可以扩；一切对外部世界透明无感知。
